03

Automatic Sensor Readings

self-hosted tier · the honest reason it exists

A static PWA has nowhere to receive an inbound push — an always-on daemon does. That asymmetry is the whole reason a self-hosted tier exists. A floating hydrometer's readings reach the batch sheet with no typing.

Device or bridge posts a reading

POST /readings

Tilt (via TiltBridge/Tilt Pi), iSpindel, RAPT (via script), or the generic shape — with a Bearer token, ideally ingest-scoped so a compromised bridge can't read or overwrite your whole brewery.

Daemon detects, normalizes & guards

Auto-detects the device shape, converts units to SG / °C, then refuses anything implausible rather than poisoning the chart. Per-device rate limit before the write.

SG 0.9–1.2temp −10–110 °CpH (0,14]≤256 KB body≥60 s / device

Resolve the device key against your links

The normalized key (e.g. tilt:RED) is matched to a batch you linked in Settings → Sensor devices. Three outcomes:

200 linked
appended

Content-addressed reading appended to the batch. Reaches every device on the next sync pull.

202 unlinked
nothing stored

Key isn't linked yet — the response tells you exactly which key to link, then Sync now.

202 batch-missing
nothing stored

Linked batch was deleted. No orphan pool by design — re-link to a live batch.

It appears on the batch sheet

Next Sync now pulls the reading into IndexedDB; use-batch-readings re-emits and it lands on the chart with a colored source badge so you can tell it from a hand-typed point.

04

Data & Sync Internals

what keeps it consistent

Single-user, multi-device — reconciled deterministically, no bespoke protocol. The sync wire payload is the backup envelope.

Source of truth

Browser Dexie / IndexedDB (v12 schema, additive migrations). Screens subscribe via liveQuery — no fetch, no server.

Repository layer

Every read and write boundary is Zod-parsed. Reactivity + validation live in src/lib/db/repos/*, not the UI.

Deletes → tombstones

A delete writes a rowTombstone in the same transaction (cascades to children). LWW by deletedAt; GC after 180 days — so a delete survives a merge instead of resurrecting.

Sync = pull · merge · push

Pull canonical, dump local, merge (LWW state + union ledger), restore, push. Deletes suppressed by tombstones; readings die with their batch.

Optimistic concurrency

ETag / If-Match on PUT /state. A 412 triggers re-pull → re-merge → retry (×3) — a lost update is structurally impossible.

Three sync modes

two-way (default), pull-only ("phone follows server"), push-only ("this desktop is canonical" — but grafts daemon-ingested readings so sensor data isn't destroyed).

Secrets never travel

Sync URL + token live in the appMeta table on the device only — never in a backup, never inside the synced payload.

BYO-AI companion

Same tool engine (src/lib/ai) runs in-app with your own API key — stored in your browser, sent only to the provider you chose.

Build & deploy

Static out/ → GitHub Pages (front-end demo), Netlify, or self-host behind Caddy + a systemd daemon. Postbuild injects a hashed CSP + SW precache.

05

Honest Edges

documented limits, not bugs

MCP writes are file-based

The AI edits an exported brewery.json, not the live database. You must re-import it in the app to see the changes — the dashed edge in §01.

Device-link lag

A sensor link saved in Settings only reaches the daemon on the next push. Until that sync lands, ingestion keeps answering 202 unlinked.

Two version axes

Dexie schema version (12) ≠ dump envelope version (10). Different axes — don't conflate them. checkSyncHealth guards the dump one.
